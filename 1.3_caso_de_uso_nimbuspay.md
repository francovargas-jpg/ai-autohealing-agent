## 1. El cliente ficticio

**NimbusPay** — fintech (empresa de tecnología financiera) que ofrece una API de pagos para comercios chicos y medianos. Procesa cobros con tarjeta vía API REST.

**Por qué justifica un agente de auto-healing (auto-reparación):**

- Cada minuto de caída de la API de pagos significa transacciones perdidas y comercios que migran a la competencia.
- El equipo de plataforma es chico (3 personas) y no tiene guardia 24/7. De madrugada, un fallo puede quedar sin atender por horas.
- El SLA (Service Level Agreement — acuerdo de nivel de servicio comprometido con el cliente) que prometen es 99.9% de disponibilidad: ~43 minutos de caída permitidos **por mes**. Un solo incidente mal resuelto se come el presupuesto entero.

Ese contexto es lo que vuelve *valioso* que un agente diagnostique y resuelva solo, en vez de esperar a que un humano despierte.

## 2. Servicios en EKS

Corren sobre **EKS** (Elastic Kubernetes Service — el Kubernetes administrado de AWS), en *deployments* separados (un deployment es el objeto de Kubernetes que mantiene N copias de un servicio vivas).

| Servicio | Qué hace | Réplicas | Criticidad |
|----------|----------|----------|------------|
| `payment-api` | Recibe los cobros, orquesta el flujo de pago | 3 | Crítico |
| `auth-service` | Valida tokens JWT, autoriza requests | 2 | Crítico |
| `notification-worker` | Manda webhooks y emails a los comercios | 2 | Medio |
| `postgres-proxy` | Pool de conexiones a la base de datos | 2 | Crítico |

Todos emiten logs a **CloudWatch** (servicio de monitoreo y logs de AWS), que es lo que el agente de diagnóstico va a leer en la Fase 03.

## 3. El fallo héroe — Deploy roto → rollback automático

Este es el escenario que se pule *end-to-end* y se demuestra en vivo. Es el que mejor cuenta la historia porque el agente **resuelve solo**, sin humanos.

### Disparador
Un desarrollador de NimbusPay sube una versión nueva de `auth-service` con un bug (ejemplo: olvidó una variable de entorno, o introdujo una excepción al arrancar). El pod entra en **CrashLoopBackOff** (estado de Kubernetes donde el contenedor arranca, crashea, se reinicia y vuelve a crashear en loop infinito). Kubernetes **nunca lo resuelve solo** — solo sigue reintentando.

### Qué hace el agente, paso a paso

```
EventBridge detecta el CrashLoopBackOff
  → Lambda dispara la Step Function
    → Agente de diagnóstico: lee logs de CloudWatch + le pregunta a Bedrock
      → Bedrock determina: "regresión de un deploy reciente, no es problema de infra"
        → Confianza ALTA
          → Agente de acción: kubectl rollout undo (revierte a la revisión sana anterior)
            → El pod vuelve a estar healthy
              → Agente de notificación: resumen por Slack
```

### Por qué es resoluble de verdad
La revisión **anterior** de `auth-service` estaba sana. El `rollout undo` (comando de Kubernetes que revierte un deployment a su versión previa) la restaura. No hay que inventar nada: la solución ya existía y el agente solo tiene que reconocer el patrón y aplicarla.

### La métrica que se mide en la demo
Tiempo desde que el pod entra en CrashLoopBackOff hasta que vuelve a estar `Ready`. **Ese número, medido en vivo, es el caso de éxito.**

## 4. Fallos secundarios soportados

No se demuestran end-to-end, pero se documentan y el agente los maneja. Sirven para ampliar la superficie de aprendizaje (más prompts a Bedrock, más decisiones) y para que el caso de referencia se vea más capaz.

| Fallo | Qué es | Qué lee el agente | Acción si confianza ALTA | Si confianza BAJA |
|-------|--------|-------------------|--------------------------|-------------------|
| **OOMKilled** | El pod excede su límite de memoria y el kernel lo mata. Kubernetes lo reinicia y vuelve a morir | Logs + métrica de memoria | Sube el `memory limit` del deployment | Abre incidente, no toca producción |
| **ImagePullBackOff** | No puede bajar la imagen Docker (tag inexistente, sin permiso en ECR) | Evento del pod: tag no encontrado | Rollback al último tag válido | Notifica con el tag culpable |
| **Readiness probe failing** | El chequeo "¿está listo para recibir tráfico?" falla; el pod corre pero no recibe requests | Logs + estado de la probe | Reinicia el pod si es transitorio | Escala a humano (probable dependencia externa caída) |

## 5. El "nivel de confianza" — corazón del diseño

Es lo que diferencia este agente de un simple script de bash con `if`. En cada fallo, **Bedrock devuelve una causa probable + un nivel de confianza** (alto / bajo). El agente actúa según ese nivel:

- **Confianza alta** → ejecuta la acción correctiva de forma autónoma.
- **Confianza baja** → **no toca producción**, escala a un humano con un resumen claro de lo que vio.

Esto le da una propiedad clave para vendérselo a Crubyt: **el agente sabe cuándo NO actuar.** Un agente que rompe producción por exceso de confianza es peor que no tener agente. Esa prudencia es parte del caso de éxito, no un detalle técnico.

## 6. Qué queda fuera de alcance (importante para un piloto)

Definir el límite ahora evita que el proyecto se infle y no llegue a la Sem 10:

- No diagnostica fallos a nivel de aplicación que requieran cambios de código (solo config y deploy).
- No maneja fallos de infraestructura de AWS (caída de una AZ, problemas de red de la VPC).
- No aprende ni se reentrena: usa Bedrock con prompts fijos, sin fine-tuning.
- Un solo cluster, una sola región. Sin multi-cluster.
